Création d'un environnement de développement sous macOS

Ainsi vous avez un Mac et voulez développer dessus...
Cela nécessite d'ajouter deux ou trois choses qui ne sont pas présentes par défaut. Veuillez également noter que les corrections de ce cours se feront exclusivement sur Linux ; il faudra donc impérativement que votre fonctionne dans un tel environnement. Pour cela nous vous proposons une infrastructure de test basée sur Docker (expliqué plus loin) que nous vous recommandons donc d'utiliser afin d'éviter toute surprise lors de la correction.

Installation des outils de développement

La première chose à installer sont les « Command Line Tools (CLT) », qui permettent l'utilisation d'outils de base tels que git.
Pour les avoir, lancez :

xcode-select --install

et vous devriez voir apparaître un dialogue d'installation. Suivez la procédure et terminez l'installation des CLT avant de passer à la suite.

Installation de « Homebrew »

La plupart des distribution de Linux contiennent un « Package Manager » permettant d'installer différents logiciels, en particuliers ceux pour le développement de code ou l'administration de la machine. Bien que macOS n'ai pas de tel « Package Manager » par défaut, nous vous recommandons d'installer et d'utiliser celui appelé « Homebrew ».
Pour l'obtenir, suivez ce lien et les instructions qui s'y trouvent.

Une fois l'installation terminée, tapez :

brew update

pour la mettre à jour.

Un bon résumé des commandes brew usuelles se trouve sous ce lien ; et il y a aussi, bien sûr, la manpage officielle.

Une fois Homebrew installé et mis à jour, avant de l'utiliser pour installer des bibliothèques (ci-dessous et plus tard), vous devez comprendre un minimum comment il procède :
quand vous installez quelque chose avec Homebrew, il met les exécutables, les fichiers d'en-tête et les bibliothèques correspondantes (décrite dans la «  Formula_ ») dans /usr/local/lib, de sorte à ne pas corrompre les éventuels logiciels similaires installé par Apple. Cela signifie que vous devez alors entrer des commandes un peu plus complexes lorsque vous souhaitez utiliser des logiciels/bibliothèques installé(e)s par brew, comme vous le verrez plus tard dans le sujet de cette semaine.

Cette légère complexité supplémentaire résulte du fait que Homebrew n'installe pas les fichiers dans les endroits connus par défaut tels que /lib ou /usr/lib. [optionnel] Si vous voulez en savoir plus sur ce que l'on appelle « rootfs » et où les différents fichiers usuels sont normalement stockés, vous pouvez lire cette page ci et cette page là. Rappelez vous cependant que macOS a une structure légèrement différente.

Bref, lorsque vous aurez à compiler du code utilisant des bibliothèques installée par brew, il faudra indiquer au compilateur d'aller chercher ces fichiers dans des répertoires différents des répertoires par défaut. Cela se fait au moyen des « flags » -I (pour la compilation) et -L (pour l'édition de liens).
Pas de panique si tout ceci est encore un peu obscure pour le moment, cela se clarifiera dans les semaines à venir, en particulier à l'aide d'exemples. Nous avons simplement regroupé l'information ici ; vous pourrez y revenir au moment opportun.

Installation de paquets logiciels

Apple fournit déjà, via les CLT que vous avez installés, des versions de make, gcc (Clang en fait !) et git.

Pour cette semaine, nous avons en plus besoin de Doxygen, que vous pouvez installer avec:

brew install doxygen

Si vous préférez utiliser le compilateur GNU CC plutôt que Clang (mais c'est Clang qui sera utilisé pour les corrections), vous pouvez aussi l'installer, avec:

brew install gcc

Ce compilateur sera alors installé dans in /usr/local/bin/gcc-x où x correspond au numéro de version installé (au moment où nous écrivons, x vaut 10).

Installation de Docker

Comme expliqué plus haut, nous utiliserons Docker pour vous fournir un environnement Linux « propre » pour tester votre code dans les mêmes conditions que notre future correction. Nous vous recommandons donc chaudement de l'installer et apprendre à l'utiliser.

Pour cela, reportez vous aux instructions données dans docker.md.
